The `inspector.json` file tells Inspector how to install dependencies and start your dev server. Place it at the root of your project.

Inspector auto-detects most projects, but `inspector.json` gives you full control — especially useful for monorepos, multi-service setups, and non-standard configurations.

## Quick start

The simplest config has one field:

```json
{
  "dev": "npm run dev"
}
```

## Let your AI agent generate it

If you're not sure what the right config is for your project, copy this prompt and paste it into your AI coding agent (Cursor, Claude Code, Copilot, etc.). The agent will analyze your project and generate the right `inspector.json`.

<Prompt
  description="Generate an **inspector.json** for this project"
  actions={["copy", "cursor"]}
>
Analyze this project and create an `inspector.json` file at the project root.

inspector.json tells the Inspector desktop app how to start the dev environment. It has this schema:

Required fields:
- "dev" (string | DevCommand[]): The dev server command(s).
  - string: a single command, e.g. "npm run dev"
  - DevCommand[]: multiple concurrent processes. Each entry is a string or { cmd, cwd } object.

Optional fields:
- "install" (string): install/setup command, runs before dev (e.g. "npm install"). Skipped on restart.
- "predev" (string | DevCommand[]): sequential commands that run every time before dev (e.g. migrations, builds).
- "cwd" (string): base working directory for all commands (relative to project root).
- "url" (string): explicit URL to connect to (bypasses stdout auto-detection)
- "build" (string): build command
- "node" (string): Node.js version constraint
- "htmlEntry" (string): main HTML file for static HTML projects

DevCommand format: string or { "cmd": "command", "cwd": "relative/path" }

Execution order: install (once) → predev (every time, sequential) → dev (concurrent if array)

Rules:
1. Look at package.json scripts, Makefile, docker-compose.yml, and any existing dev tooling.
2. If there's a single dev command, use a string.
3. If the project needs background services (databases, etc.) alongside the dev server, use an array and set "url".
4. Only include "install" if there's a meaningful install step (e.g. "pnpm install").
5. Only include "url" if the dev command won't print a localhost URL to stdout, or if dev is an array.
6. Use "predev" for commands that must complete before dev starts (e.g. db:migrate, building shared libs).
7. Use "cwd" when commands should run from a different directory than the project root.
8. Use per-command cwd ({ cmd, cwd }) when different commands need different working directories.
9. Write the file as `inspector.json` at the project root.
10. Keep it minimal — only include fields that are needed.
</Prompt>

## Scenarios

### Simple project

```json
{
  "dev": "npm run dev"
}
```

Inspector runs the command, detects the URL from stdout (e.g., `http://localhost:3000`), and connects.

### With an install step

```json
{
  "install": "npm install",
  "dev": "npm run dev"
}
```

`install` runs to completion before `dev` starts. It is skipped on dev server restart.

### Explicit URL

```json
{
  "dev": "make dev",
  "url": "http://localhost:3000"
}
```

Use `url` when the dev command doesn't print a localhost URL to stdout, or when you want to skip auto-detection entirely. This is common with orchestration tools like `make`, `docker compose`, or custom scripts.

### Multi-service (concurrent processes)

```json
{
  "dev": [
    "docker compose up postgres redis",
    "cd api && pnpm run dev",
    "cd web && pnpm run dev"
  ],
  "url": "http://localhost:3000"
}
```

When `dev` is an array, Inspector spawns all commands concurrently as long-running processes. Use `url` to tell Inspector where to connect — it can't auto-detect from multiple processes.

### Existing orchestration

```json
{
  "dev": "docker compose up",
  "url": "http://localhost:3000"
}
```

If your project already has a single command that starts everything (e.g., `docker compose`, `foreman`, `overmind`), use it as `dev` and set `url` explicitly.

### Pre-dev commands (migrations, builds)

```json
{
  "install": "pnpm install",
  "predev": [
    "pnpm run db:migrate",
    "pnpm run build:shared"
  ],
  "dev": "pnpm run dev"
}
```

`predev` commands run sequentially every time before `dev` starts. Unlike `install`, they are not skipped on restart. Use this for database migrations, building shared packages, or any setup that might change between runs.

### Per-command working directories

```json
{
  "dev": [
    { "cmd": "docker compose up", "cwd": "../infra" },
    { "cmd": "pnpm run dev", "cwd": "../api" },
    "pnpm run dev"
  ],
  "url": "http://localhost:3000"
}
```

When different services live in different directories, use the `{ cmd, cwd }` object form. Each command runs in its specified working directory. Commands without `cwd` run from the project root.

### Cross-repository setup

```json
{
  "cwd": "..",
  "install": "pnpm install",
  "predev": [
    { "cmd": "docker compose up -d", "cwd": "infra" },
    { "cmd": "pnpm run db:migrate", "cwd": "api" }
  ],
  "dev": [
    { "cmd": "pnpm run dev", "cwd": "api" },
    { "cmd": "pnpm run dev", "cwd": "web" }
  ],
  "url": "http://localhost:3000"
}
```

For projects that orchestrate multiple sibling repositories, set `cwd` to change the base directory. Per-command `cwd` values are then relative to this base.

## Working directory

By default, all commands run from the directory containing `inspector.json`. Use `cwd` to change this:

- **Top-level `cwd`**: Sets the base working directory for all commands
- **Per-command `cwd`**: Use `{ "cmd": "...", "cwd": "..." }` to set a directory for specific commands
- Per-command `cwd` is resolved relative to the top-level `cwd` (or project root if not set)
- Absolute paths override all relative resolution

## Field reference

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `dev` | Yes | `string \| DevCommand[]` | The dev server command(s). See scenarios above. |
| `install` | No | `string` | Install command. Runs before `dev`, skipped on restart. |
| `predev` | No | `string \| DevCommand[]` | Pre-dev commands. Run sequentially every time before `dev`. |
| `cwd` | No | `string` | Base working directory for all commands. |
| `url` | No | `string` | URL for Inspector to connect to. Bypasses stdout detection. |
| `build` | No | `string` | Build command. |
| `node` | No | `string` | Node.js version constraint. |
| `htmlEntry` | No | `string` | Main HTML file for static HTML projects. |

**DevCommand**: `string` or `{ "cmd": string, "cwd"?: string }`

## Execution order

1. **install** — Runs once. Skipped entirely when not provided, and skipped on dev server restart.
2. **predev** — Runs sequentially every time. Each command must complete before the next.
3. **dev** — Runs concurrently (if array) or as a single process.

## `dev` behavior by type

- **string** — Run one command. Inspector monitors the process and detects the URL from stdout.
- **DevCommand[]** — Run all commands as concurrent long-running processes. Each entry can be a string or `{ cmd, cwd }`. Inspector connects to `url`.

## Backward compatibility

Inspector reads both `inspector.json` (preferred) and the legacy `.inspector.json` (dotfile) filename. If both exist, `inspector.json` takes priority.

Older field names are also supported:

- `setup` is treated as `install`
- `run` is treated as `dev`

When saving, Inspector always writes to `inspector.json` using the new field names.
